# Karpathy 원칙 코드 리뷰

변경 사항(코드 또는 문서)이 Karpathy 4대 행동 원칙을 준수하는지 체계적으로 리뷰하는 스킬이다.
원칙 상세: `.cursor/rules/karpathy-guidelines.mdc`

## 사용 시점

- PR 리뷰 시 (QA 에이전트 또는 다른 에이전트)
- 자기 점검: 구현 완료 후 커밋 전
- 다른 에이전트의 산출물을 인수받을 때

## 리뷰 절차

### 원칙 1: 코딩 전에 생각하기

**질문:** 구현 전에 가정과 트레이드오프가 표면화되었는가?

점검 항목:
- [ ] 해석이 여러 가지 가능한 요구사항에 대해 질문하거나 대안을 제시했는가
- [ ] 가정을 명시적으로 밝혔는가 (암묵적 가정으로 진행하지 않았는가)
- [ ] 더 단순한 접근법이 있었다면 언급했는가
- [ ] 불명확한 부분에 대해 확인을 요청했는가

**위반 신호:**
- 요구사항에 없는 결정을 임의로 내림
- 여러 해석이 가능한데 하나를 묵시적으로 선택
- 트레이드오프 없이 복잡한 접근법을 바로 채택

### 원칙 2: 단순성 우선

**질문:** 문제를 해결하는 최소한의 코드/문서인가?

점검 항목:
- [ ] 요청받지 않은 기능이 추가되지 않았는가
- [ ] 일회성 코드에 불필요한 추상화가 없는가
- [ ] "미래 대비용" 유연성/설정 가능성이 추가되지 않았는가
- [ ] 코드 양이 더 줄어들 수 있는가 (200줄 → 50줄)

**위반 신호:**
- 추상 클래스/인터페이스가 구현체 1개뿐
- 설정 옵션이 요청보다 훨씬 많음
- 에러 처리가 불가능한 시나리오까지 커버
- "나중에 필요할 수 있으니" 추가된 코드

**시니어 엔지니어 테스트:** "이 코드가 과도하게 복잡하다"고 할 만한가? 그렇다면 단순화한다.

### 원칙 3: 수술적 변경

**질문:** 변경된 모든 줄이 요청으로 직접 추적 가능한가?

점검 항목:
- [ ] 인접 코드/주석/포맷을 "개선"하지 않았는가
- [ ] 깨지지 않은 것을 리팩토링하지 않았는가
- [ ] 기존 스타일과 일치하는가 (자기 선호와 다르더라도)
- [ ] 관련 없는 죽은 코드를 삭제하지 않았는가 (언급만 허용)
- [ ] 자신의 변경으로 생긴 미사용 코드는 정리했는가

**위반 신호:**
- diff에 요청과 무관한 포맷/스타일 변경이 포함
- "ついでに(겸사겸사)" 리팩토링
- 기존 코드의 주석 수정/삭제

**diff 추적 테스트:**

```bash
git diff --name-only main..HEAD
```

모든 변경 파일이 작업 범위 내인지 확인한다.

### 원칙 4: 목표 기반 실행

**질문:** 성공 기준이 정의되고 검증되었는가?

점검 항목:
- [ ] 명확한 성공 기준이 정의되어 있는가
- [ ] 성공 기준에 대한 검증이 실행되었는가
- [ ] 다단계 작업의 경우 각 단계별 검증이 있는가
- [ ] 테스트 결과가 첨부/기록되었는가

**위반 신호:**
- "구현했습니다"만 있고 검증 결과 없음
- 성공 기준이 모호함 ("잘 되게 했습니다")
- 다단계 작업에서 중간 검증 단계 누락

**계획 형식 확인:**

```
1. [단계] → 검증: [확인 사항]
2. [단계] → 검증: [확인 사항]
```

## 종합 결과 판정

| 원칙 | 통과 | 위반 시 조치 |
|------|------|-------------|
| 코딩 전에 생각하기 | 가정 명시, 대안 제시 확인 | 불명확한 가정을 명시하도록 요청 |
| 단순성 우선 | 최소 코드 확인 | 불필요한 추상화/기능 제거 요청 |
| 수술적 변경 | diff 추적 가능 | 관련 없는 변경 제거 요청 |
| 목표 기반 실행 | 검증 결과 확인 | 성공 기준 정의와 검증 실행 요청 |

## 빠른 체크리스트 (QA 에이전트용)

```markdown
## Karpathy 원칙 검증

- [ ] **코딩 전에 생각하기**: 가정이 명시되고 트레이드오프가 표면화되었는가
- [ ] **단순성 우선**: 불필요한 추상화나 추측성 코드가 없는가
- [ ] **수술적 변경**: 변경된 모든 줄이 요청으로 직접 추적 가능한가
- [ ] **목표 기반 실행**: 성공 기준이 정의되고 검증되었는가
```

## 트레이드오프 참고

이 원칙들은 **속도보다 신중함**에 무게를 둔다. 사소한 작업(오타 수정, 명확한 한 줄 변경)에는 전체 엄격함을 적용하지 않아도 된다. 비용이 큰 실수를 줄이는 것이 목표이지, 단순한 작업을 느리게 하는 것이 아니다.
